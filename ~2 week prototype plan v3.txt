## Locked Parameters

| Parameter | Value | Notes |
|---|---|---|
| Minimum post threshold | 20 posts | Production value — do not reduce. Balances statistical stability vs accessibility. |
| Minimum active post threshold | 5 active | After track routing. Gracefully rejects if fewer than 5 classify/greeting posts remain. |
| Entropy category count | 6 active buckets | Collapse rare buckets into "Other" |
| Sub-window entropy gate | 60 posts | Below this → frequency variance only |
| Epoch window | 30 days | Updated from original 7-day spec. More accounts reach the 20-post threshold. Consistency pillar switches to thirds_entropy mode at 60+ posts. |
| Anchor chain | Avalanche C-Chain | EVM-compatible, L1 path available later |
| Domain | tessera.xyz | |
| LLM temperature | 0 | Non-negotiable for score determinism |
| Prompt version | v5 | Hash: `1deff518231c…` — frozen. Accuracy: 84%. |

### Day 4 — Attestation Layer 🔜 NEXT

**Sealing is automatic. The user's only act is claiming.**

```
Claim registered
      ↓
Weekly cron (Sunday midnight UTC)
      ↓
Pull → Score → Canonical Hash → Anchor Onchain
      ↓
Store TX hash, update status: sealed
      ↓
Post notification reply to user tagging them
```

**Architecture decisions (locked):**
- Protocol wallet pays all gas — users never sign transactions in v1
- Deploy contract once to Avalanche C-Chain mainnet — reuse across all epochs
- One `anchor()` call per epoch per user — no batching for prototype
- Contract emits event only — no storage, no access control, no upgrades

**Smart contract — Solidity:**
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract TesseraAnchor {
    event Anchored(address indexed sender, bytes32 indexed snapshotHash, uint256 timestamp);

    function anchor(bytes32 snapshotHash) external {
        emit Anchored(msg.sender, snapshotHash, block.timestamp);
    }
}
```

**Deploy script — Hardhat / ethers.js targeting Avalanche C-Chain:**
- Network: Avalanche C-Chain mainnet (chainId 43114, RPC: https://api.avax.network/ext/bc/C/rpc)
- Store deployed contract address in config — never hardcode inline
- Verify on Snowtrace after deploy

**Python sealer (`attestation/sealer.py`):**
```python
# anchor_epoch(snapshot: dict, private_key: str, contract_address: str) -> dict
# Returns: { tx_hash, block_number, anchored_at }
```
- Load contract ABI from file (not hardcoded)
- Connect via web3.py to Avalanche C-Chain
- Build and sign transaction from protocol wallet
- Submit and wait for receipt (1 confirmation sufficient for prototype)
- Return tx_hash, block_number, anchored_at timestamp

**Cron (`attestation/cron.py`):**
- Runs weekly, Sunday midnight UTC
- Fetches all accounts with `status=computed`
- For each: fetch cached collection → score → seal → store tx_hash + block_number → update status to `sealed`
- Post Arena notification to user: *"Epoch [start–end] sealed. TX: 0x... · tessera.xyz/handle"*
- On failure: log, mark `status=seal_failed`, retry next run — never silently drop

**Cron resilience:**
- [ ] Store `last_seal_run` timestamp in a state file (pre-DB) or DB row
- [ ] On startup: if `last_seal_run` > 7 days ago, execute immediately
- [ ] Idempotent: before every `anchor()` call, check if epoch already has a tx_hash — skip if so
- [ ] Alert/log if cron has not run within 25 hours

**Day 4 checklist:**

### Day 5 — Data Layer + Internal API

**Supabase schema:**

```sql
-- users: one row per claimed handle
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    handle TEXT UNIQUE NOT NULL,
    arena_user_id TEXT NOT NULL,
    claimed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_epoch_id UUID REFERENCES epochs(id)
);

-- epochs: one row per scoring window per handle
CREATE TABLE epochs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    handle TEXT NOT NULL,
    epoch_start TIMESTAMPTZ NOT NULL,
    epoch_end TIMESTAMPTZ NOT NULL,
    status TEXT NOT NULL CHECK (status IN ('computed', 'sealed', 'seal_failed')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- scores: pillar scores for each epoch
CREATE TABLE scores (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    epoch_id UUID NOT NULL REFERENCES epochs(id),
    originality NUMERIC(5,2),
    focus NUMERIC(5,2),
    consistency NUMERIC(5,2),
    depth NUMERIC(5,2),
    composite NUMERIC(5,2),
    methodology TEXT NOT NULL DEFAULT 'v1.0',
    snapshot_json JSONB,        -- full canonical snapshot stored here
    collection_hash TEXT,
    prompt_hash TEXT,
    model TEXT
);

-- anchors: onchain seal receipts
CREATE TABLE anchors (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    epoch_id UUID NOT NULL REFERENCES epochs(id),
    tx_hash TEXT NOT NULL,
    block_number BIGINT NOT NULL,
    snapshot_hash TEXT NOT NULL,
    anchored_at TIMESTAMPTZ NOT NULL
);

-- notifications: polling crash recovery
CREATE TABLE bot_state (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Usage: INSERT INTO bot_state (key, value) VALUES ('last_processed_notification_id', '...')
--        ON CONFLICT (key) DO UPDATE SET value = EXCLUDED.value;

-- command_log: rate limiting + abuse detection
CREATE TABLE command_log (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    handle TEXT NOT NULL,
    command TEXT NOT NULL,
    issued_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX ON command_log (handle, issued_at);
```

**Internal API endpoints (FastAPI):**
- `POST /api/score` — trigger scoring for a handle (uses cache if available)
- `GET /api/score/:handle` — retrieve epoch history
- `GET /api/audit/:epoch_id` — full audit trail including snapshot_json
- `POST /api/cron/seal` — weekly cron trigger (called by scheduler, not user)
- `POST /api/bot/command` — receive and route bot commands (Day 6)

### Day 6 — Bot Command Listener + Router

**Polling loop — built for failure, not success**

The polling loop will crash. Plan for it from the start.

- [ ] Poll `GET /agents/notifications/unseen` every 60 seconds
- [ ] Store `last_processed_notification_id` in `bot_state` table after each batch
- [ ] On restart: load last processed ID, re-check unseen from that point
- [ ] Idempotent command handlers — processing the same notification twice must produce the same result
- [ ] Backoff: if 3 consecutive poll failures, wait 5 minutes before retrying
- [ ] Alert/log if polling loop silent for > 5 minutes

**Command parser:**
- [ ] Filter for `mention` type notifications only
- [ ] Strip `@bannerusmaximus`, extract first keyword, case-insensitive
- [ ] Parse `inspect` target handle from remainder of post
- [ ] Unknown commands → reply listing the 3 available commands
- [ ] Rate limiting: max 5 commands per user per hour (check `command_log`)
- [ ] Global cap: max 30 commands per minute across all users
- [ ] Cache `inspect` results for 10 minutes — don't re-score on spam
- [ ] Epoch cache: if epoch window hasn't changed, return stored snapshot — never recompute, never re-run LLM

**Bot reply rule:** Always post a new thread tagging the user. Never reply to old threads.

## What to Cut If You're Behind

In this order only:

1. **tessera.xyz full site** — single static audit page is enough
2. **`reveal` command** — `claim` and `inspect` are the two that matter most for demo
3. **Supabase** — JSON files on disk, proper DB after demo
4. **Canonical hash verification on audit page** — show the hash, add reproduce-it instructions later

**Never cut:**
- `inspect` — the viral mechanic
- The audit page — the answer to "why is this credibly neutral?"
- The auto-seal cron — without it the protocol has no pulse
- LLM determinism controls — without these, scores drift and the whole thing is untrustworthy

## Risk Register

| Risk | Mitigation |
|---|---|
| LLM output drift between runs | temperature=0, prompt v5 frozen (hash `1deff518231c…`), determinism confirmed |
| Entropy fragile at low post counts | 6-bucket taxonomy, 20-post minimum, documented v1.0 limitation |
| Quote-heavy accounts with very few classifiable posts | 5 active post minimum after routing, clean rejection message |
| Canonical hash mismatch | `sort_keys=True`, fixed separators, UTF-8, scores locked at 2dp, timestamps ISO 8601 |
| Polling loop crashes silently | Store last processed ID in DB, backoff logic, silence alerting |
| Cron misses a run | Store `last_seal_run`, execute immediately on restart if missed, alert if silent > 25 hours |
| Double-seal on restart | Cron idempotent — checks tx_hash in anchors table before every onchain write |
| `inspect` spam exhausting rate limits | Cache 10 minutes, global 30/min cap |
| Depth pillar challenged | v1.0 definition explicitly documented in audit view |
| Minimum threshold excludes users | 20 posts at 6 buckets is defensible — document tradeoff |
| Gas costs on Avalanche | ~$0.01–0.02 per seal. 100 users/week = ~$1–2. Not a concern for prototype. |
| Private key exposure | Never commit to repo — load from environment variable only |
